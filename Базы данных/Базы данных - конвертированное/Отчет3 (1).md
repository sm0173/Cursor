# 1. Цель и состав выполненной работы

В рамках задания №3 выполнена физическая реализация базы данных для складского контура интернет-магазина. Основная задача этапа состояла в том, чтобы перевести логическую модель в физическую модель PostgreSQL, создать все объекты базы данных, настроить ключи и ограничения, а затем подтвердить корректность реализации на тестовых сценариях.

На этапе 3 были решены следующие практические задачи:

• выбран набор типов данных PostgreSQL для хранения идентификаторов, кодов, наименований, количественных показателей, статусов и временных меток;

• подготовлена физическая модель в нотации IDEF1X для схемы ecommerce_wms;

• созданы таблицы, связи PK-FK, уникальные ограничения и проверки целостности;

• выполнено тестирование операций INSERT, SELECT и DELETE на тестовых данных;

• отдельно проверены недопустимые сценарии: нарушение внешнего ключа, CHECK-ограничения и уникальности.

# 2. Краткая характеристика физической модели

База данных реализована в схеме ecommerce_wms и охватывает три основные области: справочники и складскую структуру, обработку заказов и операции исполнения, а также контроль, инциденты и возвраты.

## 2.2. Выбор типов данных и средств ограничения допустимых значений

При реализации структуры использованы как стандартные типы PostgreSQL, так и пользовательские ENUM-типы. Такой подход позволяет одновременно обеспечить понятную модель данных и встроенную валидацию значений на уровне СУБД.

| Тип данных | Где используется | Обоснование |
| --- | --- | --- |
| bigserial | Первичные ключи | Уникальный суррогатный идентификатор таблиц, удобный для связей и масштабирования. |
| bigint | Внешние ключи | Тип совместим с bigserial, поэтому связи PK-FK реализованы без конфликтов типов. |
| character varying(50/100/150/255) | Коды и наименования | Используется для SKU, штрихкодов, кодов ячеек, названий складов и товаров. |
| numeric(10,3) | Весовые характеристики | Подходит для unit_weight и expected_total_weight, где важны дробные значения. |
| integer | Количественные показатели | Используется для ordered_qty, collected_qty, quantity_on_hand. |
| timestamp without time zone | Даты и время операций | Фиксирует моменты создания задания, операций, инцидентов и т.д. |
| boolean | Признак активности | Используется, например, в product.is_active и employee.is_active. |
| ENUM types | Статусы и классификаторы | Пользовательские типы ограничивают допустимые значения статусов и видов операций. |

### Примеры пользовательских ENUM-типов, найденных в физической модели:

• barcode.barcode_type -> barcode_type_enum

• corrective_action.action_type -> corrective_action_type_enum

• incident.incident_type -> incident_type_enum

• incident.detected_stage -> detected_stage_enum

• incident.status -> incident_status_type

• order.order_status -> order_status_type

• packing.packing_status -> packing_status_type

• picking_operation.operation_result -> operation_result_type

• picking_task.task_status -> task_status_type

• qc_check.qc_result -> qc_result_type

## 2.3. Соблюдение требований 3НФ

• Каждая таблица содержит атомарные атрибуты, а повторяющиеся группы вынесены в отдельные сущности.

• Зависимости между объектами разложены на отдельные таблицы-справочники: роли сотрудников, склады, зоны и ячейки хранения представлены раздельно.

• Нестабильные бизнес-атрибуты не дублируются между таблицами: например, сведения о товаре находятся в product, а строки заказа содержат только ссылки на товар и количественные показатели.

• Операционные сущности (picking_task, picking_task_item, picking_operation, scan_event) разделены по уровням детализации, поэтому каждая таблица хранит данные только своего процесса.

• Инциденты, корректирующие действия и возвраты также выделены в самостоятельные сущности, что исключает транзитивные зависимости и дублирование.

## 3.2. Недопустимые сценарии


Рисунок 7 - Недопустимый сценарий: ссылка на несуществующий заказ.

Получена ошибка SQLSTATE 23503: нарушено ограничение внешнего ключа fk_order_item_order, поскольку order_id = 9999 отсутствует в таблице "order".


Рисунок 8 - Недопустимый сценарий: пустое наименование склада.

Получена ошибка SQLSTATE 23514: нарушено CHECK-ограничение chk_warehouse_name_not_blank. Это подтверждает валидацию обязательных текстовых атрибутов.


Рисунок 9 - Недопустимый сценарий: дублирование активного задания на один заказ.

Получена ошибка SQLSTATE 23505: нарушено ограничение уникальности uq_picking_task_active_order. Тем самым подтверждена защита от дублей активных задач.
