ЗАДАНИЕ №3.


к групповому проекту по учебной дисциплине «Базы данных»


Физическая реализация базы данных


Выберите подходящие типы данных СУБД PostgreSQL для хранения сведений в базе данных в соответствие с логической моделью и описанными ограничениями.

Спроектируйте физическую модель данных с учетом требований 3НФ (третьей нормальной формы). Результат проектирования необходимо представить в виде диаграммы в нотации IDEF1X.

Создайте базу данных на сервере PostgreSQL и реализуйте объекты в соответствие с физической моделью.

Протестируйте реализацию с помощью запросов на выставку/обновление/удаление на тестовых данных (достаточно 1-2 записи для каждой таблицы). Тестирование должно охватывать все сценарии (допустимые и недопустимые данные).


В результате выполнения задания, должны быть приложены:

- текст пояснительной записки, включающей результаты выполнения заданий (все описания, схемы, пояснения, скриншоты с результатами тестирования) на групповой проект, включая задание 1 и задание 2 (файл docx, pdf);

- выгрузку БД (backup);

- скрипты DDL (файл .sql);

- скрипты DML (файл .sql).

Если объект БД создается в визуальном режиме, то нужно приложить скрипт, генерируемый автоматически.


Критерии оценки задания (используются при расчете балла по критерию K2 оценки группового проекта в программе учебной дисциплины).


K2.3.1 – Типы данных выбраны корректно (позволяют обеспечить валидность данных)

K2.3.2 – Спроектированная база данных удовлетворяет требованиям 3НФ

K2.3.3 – Диаграмма физической модели данных выполнена с соблюдением правил нотации IDEF1X, не содержит синтаксических и смысловых ошибок

K2.3.4 – Реализация базы данных полностью соответствует физической модели (созданы все необходимые объекты, структура объектов соответствует проекту)

K2.3.5 – Корректно и полно реализованы первичные и внешние ключи

K2.3.6 – Корректно и полно реализованы все прочие ограничения (за исключением особо сложных случаев, которые планируется реализовать на следующих стадиях проекта, о чем указано в пояснениях)

K2.3.7 – Реализовано полное тестирование разработки (учтены допустимые и недопустимые сценарии)

K2.3.8 – Описание реализации включает все необходимые пояснения по выбору решений и комментарии по результатам тестирования.


Пояснения к заданию.


При выполнении заданий используйте материалы лекций 2, 3, 4.

Пример описания таблицы базы данных (для п.1).


Название таблицы: meeting

| Имя атрибута | Тип данных | Ограничение | Пояснение |
| --- | --- | --- | --- |
| meeting_id | bigserial | PK | *Искусственный первичный ключ (простой). Выбор длинного целого обусловлен спецификой работы компании (большое количество бронирований в день). |
| meeting_period | tsrange | check | **Проверка на вхождение в диапазон 9:00 – 18:00 |
| room_num | smallint | FK | ссылка на справочник помещений |
| … |  |  |  |

Примечания.

* Если будет использован другой тип PK, то нужно обосновать.

**Данная проверка является сложной и может быть реализована на следующих этапах


Для реализации ограничений рекомендуется использовать проверки check, пользовательские типы данных Types и ограничения типа исключений exclude.

Примеры использования проверок (check): время встречи только в рабочие часы (с 9:00 до 18:00); время окончания встречи позже времени начала; плановое значение KPI сотрудника не ниже порогового.

Примеры использования пользовательских типов данных (Types): статус встречи – назначена, завершена, отменена; тип встречи – внутренняя, с клиентом; оценка состояния оборудования – неисправно, удовлетворительное, отличное.

Примеры использования ограничений типа исключений (exclude): проверка на пересечение временных интервалов при планировании встреч для одного и того же сотрудника; проверка на пересечение временных интервалов при планировании встреч в одной комнате.
