# Отчёт по заданию №4

**Дисциплина:** «Базы данных».  
**Тема проекта:** учёт комплектации заказов на складе (e-commerce).  
**СУБД:** PostgreSQL. **Схема:** `ecommerce_wms`.

---

## 1. Цель работы

По заданию №4 необходимо:

1. Заполнить базу **тестовыми данными** в объёме, достаточном для проверки бизнес-логики.
2. Реализовать **бизнес-логику на SQL**: запросы, **представления (VIEW)**, **пользовательские функции** в соответствии с ранее сформулированными функциональными требованиями (FR из задания 1).
3. **Протестировать** реализацию (в скрипте — контрольные выборки).

В качестве основы используется база, спроектированная и созданная на этапе 3 (DDL, ограничения, связи). На этапе 4 новые таблицы не вводились, добавлены только объекты логики и данные для проверки.

---

## 2. Исходные условия

- Перед выполнением скрипта этапа 4 база должна быть **восстановлена из резервной копии** после этапа 3 (как требует цепочка заданий).
- Все тестовые сущности помечены префиксами **`stage4-`**, **`stage4_`** (имена складов, зон, ячеек, SKU, внешних номеров заказов, ролей и т.д.), чтобы не перепутать их с возможными учебными данными других этапов.
- Таблица заказов в PostgreSQL называется `order` — это зарезервированное слово, поэтому в запросах она указывается как **`"order"`** (в кавычках). Если кавычки забыть, скрипт выдаст ошибку.

## 4. Логика скрипта `ecommerce_wms_stage4_dml.sql`

Скрипт рассчитан на выполнение **одним запуском** в pgAdmin (Execute/ F5 на всём файле). Порядок такой:

1. Установка кодировки клиента `UTF8`.
2. Создание или замена **представлений** (`CREATE OR REPLACE VIEW`).
3. Создание или замена **функций** (`CREATE OR REPLACE FUNCTION`).
4. Вызов функции **`stage4_seed_demo_data()`**, которая вставляет тестовые строки во все нужные таблицы с учётом внешних ключей.
5. Выполнение **контрольных SELECT** в конце файла: у каждого запроса в результате есть колонка **`ok`** (логическое значение). Ожидается **`true`** по всем строкам.

Если данные этапа 4 уже загружались (существует заказ с `external_order_no = 'stage4-ext-001'`), функция наполнения выводит **NOTICE** и завершает работу без повторной вставки — это защита от дублирования при случайном втором запуске.

## 5. Реализованные представления (VIEW)

### 5.1. `ecommerce_wms.stage4_vw_order_detail`

**Назначение:** показать по тестовым заказам состав заказа: связка заказ — позиции — товар (SKU, наименование), количества, статус заказа.

**Идея:** один заказ может иметь несколько строк в `order_item`; представление «разворачивает» заказ в плоский вид, удобный для проверки и для отчёта.

**Ограничение выборки:** только заказы, у которых `external_order_no LIKE 'stage4-ext-%'`, чтобы не смешивать с чужими данными.

### 5.2. `ecommerce_wms.stage4_vw_incident_return`

**Назначение:** по тем же тестовым заказам вывести связку **инцидент ↔ возврат** (если записи есть). Используются `LEFT JOIN`, потому что у «успешного» заказа инцидента и возврата может не быть.

Это приближает требования блока **R5** (возвраты, причины, связь с заказом и инцидентом) и часть **R1** (ошибки как инциденты).

## 6. Реализованные функции

### 6.1. `stage4_order_line_count(p_order_id bigint)` → `bigint`

Язык **`LANGUAGE sql`**. Считает число строк в `order_item` для указанного заказа. Небольшая вспомогательная функция.

### 6.2. `stage4_order_health(p_order_id bigint)` → `text`

Язык **`LANGUAGE plpgsql`**. Внутри:

- вызывается **`stage4_order_line_count`** (так выполняется требование методички о **вызове одной функции из другой**);
- проверяется наличие **открытого** инцидента (`status` в `open` / `in_progress`);
- возвращается короткая **текстовая сводка**: сколько позиций в заказе и есть ли открытый инцидент.

Удобно для быстрой проверки «в каком состоянии заказ» без ручного разбора нескольких таблиц.

### 6.3. `stage4_order_timeline(p_order_id bigint)` → таблица `(event_ts, event_kind, event_detail)`

Язык **`LANGUAGE sql`**. Формирует **упрощённую хронологию** по заказу: дата/время события, тип события, краткое описание (статус или результат). Источники объединяются через **`UNION ALL`**: заказ, завершённое задание на отбор, операции отбора, весовой контроль, контроль качества (QC).

Это отвечает идее **«карточки заказа»** по времени (FR **R2.3**) в учебном упрощённом виде: не все возможные события модели включены в union, только те, по которым в тестовых сценариях гарантированно есть данные.

### 6.4. `stage4_seed_demo_data()` → `void`

Язык **`LANGUAGE plpgsql`**. Содержит последовательность **`INSERT`** в справочники и операционные таблицы с соблюдением порядка внешних ключей. Так выполняется часть задания про **заполнение данными** и про **вставки в теле функции** (как один из допустимых вариантов из пояснений к заданию 4).

## 7. Тестовые сценарии данных

### 7.1. Заказ `stage4-ext-001` («нормальный» путь)

- Два SKU в составе заказа, количества заказано = собрано.
- Задание на отбор в статусе **завершено**, операции отбора с результатом **успех**, есть **успешное** сканирование штрихкода.
- Весовой контроль **пройден**, QC **пройден**, упаковка **packed**, статус заказа **shipped**.
- **Открытых инцидентов нет** — для проверки функции `stage4_order_health`.

### 7.2. Заказ `stage4-ext-002` («ветка с ошибкой»)

- Одна позиция; операция отбора с результатом **partial**, скан **неуспешный**.
- Заведён **инцидент** (тип `wrong_item`, этап `picking`, статус **in_progress**), есть **корректирующее действие**.
- Весовой контроль **не пройден**, QC — **на доработку**, упаковка **repacked**.
- Оформлен **возврат** с привязкой к инциденту.

Таким образом в одной базе одновременно есть «чистый» и «проблемный» пример для отчётов и для контрольных запросов.

## 8. Дополнительные SQL-конструкции

В конце файла, помимо проверок с колонкой `ok`, есть запрос с **общим табличным выражением (CTE)** **`WITH x AS (...)`** и агрегированием **`GROUP BY`** по типу инцидента. Это иллюстрирует рекомендации из пояснений к заданию (CTE, вложенная логика) и привязку к FR **R1.4 / R5.2** в минимальном виде (счётчики по типам для тестовых заказов).

## 9. Порядок проверки (тестирование)

1. Убедиться, что подключение идёт к нужной базе и схеме **`ecommerce_wms`** существует.
2. При необходимости **очистить** тестовые данные этапа 4 (удаление в обратном порядке FK или восстановление БД из бэкапа этапа 3).
3. Выполнить **`ecommerce_wms_stage4_dml.sql`** целиком.
4. В панели результатов пролистать **нижние запросы**: везде **`ok = true`**.
5. Для наглядности в отчёт к преподавателю можно приложить скриншоты из pgAdmin, например:
   - выборка из **`stage4_vw_order_detail`**;
   - результат **`stage4_order_timeline`** для заказа `stage4-ext-001`;
   - текст, возвращаемый **`stage4_order_health`**, для обоих тестовых заказов.

## 10. Ограничения и допущения

- **FR R4.3 (фотофиксация):** в ER-модели после этапа 3 **отдельной таблицы под фото не предусмотрено**, поэтому отдельная реализация на SQL не выполнялась; в матрице FR это отмечено явно.
- **R3.3** (расширенная аналитика по производительности) **не выносилась в отдельную функцию**: по смыслу задания 4 достаточно демонстрации логики на представлениях, функциях и агрегирующем запросе; при запросе преподавателя можно добавить отдельную функцию или представление.
- Повторный прогон скрипта **без очистки** не создаёт второй комплект данных благодаря проверке в начале `stage4_seed_demo_data`.
